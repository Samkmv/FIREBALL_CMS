# FIREBALL CMS 1.7.0

FIREBALL CMS - PHP CMS для блогов, информационных порталов, корпоративных сайтов, баз знаний и расширяемых веб-проектов. Система построена вокруг MVC-ядра, административной панели, тем оформления, плагинов, локализации, PWA-инфраструктуры и центра обновлений.

Версия 1.7.0 подготовлена как стабильный релиз: основной фокус сделан на надежности, совместимости, безопасности, документации и готовности к production-использованию.

## Описание проекта

FIREBALL CMS предоставляет готовую административную панель, публичную часть, установщик, управление контентом, файловый менеджер, систему тем, систему плагинов, аналитику, уведомления, PWA и инструменты обслуживания.

Архитектура сохраняет разделение ядра, тем и плагинов:

* ядро отвечает за маршрутизацию, MVC, безопасность, настройки, обновления и системные сервисы;
* темы отвечают за публичный внешний вид и используют Theme API;
* плагины расширяют CMS через PluginManager, хуки, события, маршруты, миграции и настройки.

## Требования

* PHP 8.2 или новее;

* Apache с rewrite или Nginx с маршрутизацией на `public/index.php`;
* HTTPS для production и PWA Push;
* Composer для установки PHP-зависимостей.

### PHP-расширения

* PDO;
* pdo_mysql;
* mbstring;
* json;
* fileinfo;
* openssl;
* gd или imagick для обработки изображений;
* curl рекомендуется для обновлений и внешних запросов.

## Установка

1. Скопируйте проект на сервер.
2. Настройте корень сайта на директорию `public/`.
3. Установите зависимости:

```bash
composer install
```

4. Убедитесь, что директории `storage/`, `public/uploads/`, `tmp/` и пользовательские директории тем доступны для записи веб-серверу.
5. Откройте сайт в браузере.
6. Если CMS не установлена, откроется мастер `/install`.

Мастер установки выполняет:


## Работа с темами

Темы находятся в `themes/{theme-slug}/`.

Типовая структура темы:

```text
themes/example/
  theme.json
  templates/
  partials/
  assets/
```

`theme.json` описывает тему, версию, автора и поддерживаемые возможности. Публичные шаблоны получают данные через Theme API и helper-функции CMS. Тема не должна напрямую менять ядро CMS.

Рекомендуемый процесс:

1. создать копию системной темы;
2. изменить `theme.json`;
3. редактировать шаблоны через редактор тем или файловую систему;
4. проверять PHP-синтаксис и структуру `theme.json`;
5. использовать preview перед активацией;
6. хранить пользовательские изменения отдельно от системной темы.

## Работа с плагинами

Плагины находятся в `plugins/{plugin-slug}/`.

Минимальная структура:

```text
plugins/example-plugin/
  plugin.json
  Plugin.php
  routes.php
  admin.php
  migrations/
  views/
  assets/
  lang/
```

`plugin.json` должен содержать `name`, `slug`, `version`, `description`, `author` и требования к CMS.

Плагин может:

* регистрировать пункты меню;
* подключать маршруты;
* выполнять миграции;
* хранить настройки в `plugin_settings`;
* использовать хуки и события;
* подключать собственные переводы;
* добавлять административные и публичные страницы.

Плагины не должны изменять файлы ядра CMS. Все интеграции выполняются через PluginManager, хуки, события, маршруты и настройки.

## Локализация

Системные языковые файлы находятся в `app/Languages/`.

Публичные языковые файлы могут находиться в `public/lang/`.

Плагины должны хранить свои строки отдельно:

```text
plugins/{plugin-slug}/lang/ru.php
plugins/{plugin-slug}/lang/en.php
plugins/{plugin-slug}/lang/de.php
plugins/{plugin-slug}/lang/zh-cn.php
```

Для новых строк используйте существующий helper перевода и уникальные ключи пространства плагина или модуля.

## Обновление CMS

Текущая версия описывается в `config/version.php` и `update.json`.

Центр обновлений поддерживает:

* проверку новых версий;
* каналы `Стабильная версия` и `Для разработчиков`;
* установку обновлений;
* защиту пользовательских директорий;
* backup перед обновлением;
* выполнение миграций;
* откат через Git, если сайт установлен из Git-репозитория и доступен исходный commit.

Перед обновлением production-сайта рекомендуется:

1. создать backup базы данных;
2. создать backup файлов;
3. проверить совместимость темы и плагинов;
4. выполнить обновление сначала на staging-копии.

## Разработка

Полезные команды:

```bash
composer install
composer dump-autoload
php -l path/to/file.php
```

Рекомендации:

* не менять Theme API без необходимости;
* не добавлять бизнес-логику в шаблоны;
* не подключать сторонние UI-наборы без причины;
* использовать CSRF для POST-действий;
* валидировать входные данные на сервере;
* экранировать пользовательский вывод;
* хранить расширения в плагинах;
* хранить изменения внешнего вида в темах;
* проверять мобильную версию админки и публичной части.
