# FIREBALL CMS Update Cheatsheet

## Главное правило перед релизом

Перед каждым `commit` на релиз обязательно обновляйте файл [config/version.php](/Applications/MAMP/htdocs/FIREBALL_CMS/config/version.php:1).

Что менять в `config/version.php`:

- `version` — новая версия, например `1.5.2`
- `released_at` — дата релиза в формате `YYYY-MM-DD`
- `summary` — короткое описание релиза
- `changes` — список основных изменений

Без этого в админке и в данных обновления будет показываться старая версия.

## Как выпускать обновление у себя

1. Внести и проверить все изменения локально.
2. Обновить [config/version.php](/Applications/MAMP/htdocs/FIREBALL_CMS/config/version.php:1).
3. Открыть GitHub Desktop.
4. Проверить список изменённых файлов.
5. Убедиться, что в коммит не попали лишние файлы.
6. Сделать `Commit to main`.
7. Нажать `Push origin`.

## Как выложить релиз на GitHub

1. Открыть репозиторий на GitHub.
2. Перейти в `Releases`.
3. Нажать `Draft a new release`.
4. Создать новый tag, например `v1.5.2`.
5. В `Target` выбрать свежий коммит из `main`.
6. В `Release title` указать ту же версию, например `v1.5.2`.
7. В описании перечислить основные изменения.
8. Скачать свежую сборку с Github и прикрепить ее в `Releases` с названием и номером версии, например `v1.5.2`.
9. Нажать `Publish release`.

Важно:

- Клиент обновляется из GitHub `release/tag`, а не из вашей локальной папки.
- Если изменения есть только локально и не выпущен новый release, клиент их не получит.

## Что делать у клиента

### Первый запуск

1. Установить сайт из чистого архива GitHub Release.
2. Переименовать `config.local.example.php` в `config/config.local.php`
2. Настроить `config/config.local.php` под сервер клиента.
3. В админке открыть страницу `Обновление`.
4. Указать GitHub-репозиторий в формате `owner/repo`.
5. При необходимости указать GitHub token.
6. Сохранить настройки.

### Проверка обновления

1. Открыть страницу `Обновление`.
2. Нажать `Проверить обновления`.
3. Если найден новый релиз, нажать `Обновить`.

## Что именно менять у клиента

У клиента обычно меняется только:

- серверный `config/config.local.php`
- база данных
- пользовательские файлы в `public/uploads`

Это не должно храниться в вашем релизе.

Не надо перезаписывать у клиента:

- `config/config.local.php`
- содержимое `public/uploads`
- временные файлы из `tmp`

## Какой архив давать клиенту

Правильно:

- давать ZIP из опубликованного GitHub Release

Неправильно:

- архивировать локальную папку через Finder
- давать клиенту локальную версию, которой нет в GitHub

Почему:

- GitHub Release совпадает с тем, откуда updater будет скачивать следующие обновления
- локальный архив легко загрязняется `__MACOSX`, `.DS_Store` и лишними файлами

## Как повышать версию

- `1.5.2` — мелкие исправления
- `1.6.0` — новые функции без ломающих изменений
- `2.0.0` — ломающие изменения

## Мини-чеклист перед публикацией

1. Изменения завершены.
2. [config/version.php](/Applications/MAMP/htdocs/FIREBALL_CMS/config/version.php:1) обновлён.
3. Проверены нужные файлы в GitHub Desktop.
4. Сделан `commit`.
5. Сделан `push`.
6. Выпущен новый GitHub Release.
7. У клиента на странице `Обновление` указан правильный репозиторий.

## Мини-чеклист перед обновлением у клиента

1. Есть свежий опубликованный Release на GitHub.
2. Репозиторий в настройках обновления заполнен правильно.
3. На сервере клиента доступен GitHub.
4. У проекта есть права на запись.
5. Нажать `Проверить обновления`.
6. Нажать `Обновить`.
