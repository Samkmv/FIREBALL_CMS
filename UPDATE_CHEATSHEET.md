# FIREBALL CMS Update Cheatsheet

## Главное правило перед релизом

Перед каждым `commit` на релиз обязательно обновляйте файл [config/version.php](/Applications/MAMP/htdocs/FIREBALL_CMS/config/version.php:1).

Что менять в `config/version.php`:

- `version` — новая версия, например `1.5.2`
- `released_at` — дата релиза в формате `YYYY-MM-DD`
- `summary` — короткое описание релиза
- `changes` — список основных изменений

Без этого в админке и в данных обновления будет показываться старая версия.

## Как выпускать обновление у себя

1. Внести и проверить все изменения локально.
2. Обновить [config/version.php](/Applications/MAMP/htdocs/FIREBALL_CMS/config/version.php:1).
3. Открыть GitHub Desktop.
4. Проверить список изменённых файлов.
5. Убедиться, что в коммит не попали лишние файлы.
6. Сделать `Commit to main`.
7. Нажать `Push origin`.

## Как подготовить обновление в репозитории

1. Открыть репозиторий на GitHub.
2. Убедиться, что нужные изменения уже попали в рабочую ветку, например `main`.
3. При необходимости создать новый tag, например `v1.5.2`.
4. Сделать `Push origin`.

Важно:

- Клиент обновляется из выбранной GitHub-ветки.
- Если установка развернута как git-репозиторий, updater использует `git fetch` и `git pull --ff-only`.
- Если сайт установлен из ZIP без `.git`, updater скачает ZIP-архив исходников этой же ветки и обновит файлы поверх установки.
- Если изменения есть только локально и не запушены в удалённую ветку, клиент их не получит.
- GitHub Release теперь не обязателен для автообновления. Он нужен только если вы хотите вести отдельную историю релизов.

## Что делать у клиента

### Первый запуск

1. Развернуть сайт через `git clone` или из ZIP-архива.
2. Создать `config/config.local.php` на основе `config/config.local.example.php`.
3. Настроить `config/config.local.php` под сервер клиента.
4. Выполнить `composer install`.
5. В админке открыть страницу `Обновление`.
6. Указать GitHub-репозиторий в формате `owner/repo`.
7. Указать ветку обновления, обычно `main`.
8. При необходимости указать GitHub token.
9. Сохранить настройки.

### Проверка обновления

1. Открыть страницу `Обновление`.
2. Нажать `Проверить обновления`.
3. Если удалённая ветка или metadata `config/version.php` показывают новую версию, нажать `Обновить`.

## Что именно менять у клиента

У клиента обычно меняется только:

- серверный `config/config.local.php`
- база данных
- пользовательские файлы в `public/uploads`

Это не должно храниться в вашем репозитории.

Не надо перезаписывать у клиента:

- `config/config.local.php`
- содержимое `public/uploads`
- временные файлы из `tmp`

## Как разворачивать клиенту

Правильно:

- если есть доступ к git, клонировать репозиторий на сервер клиента
- если git недоступен, можно развернуть сайт из ZIP
- держать сайт на нужной ветке, например `main`
- обновлять код через встроенный updater

Неправильно:

- менять tracked-файлы прямо на production git-установке
- держать на сервере ветку, которая не совпадает с веткой обновления в админке
- перезаписывать клиентские `config/config.local.php` и `public/uploads`

Почему:

- git-установка обновляется безопасным `fast-forward`
- ZIP-установка тоже может обновляться автоматически без `.git`
- клиентские файлы и локальный конфиг сохраняются при ZIP fallback

## Как повышать версию

- `1.5.2` — мелкие исправления
- `1.6.0` — новые функции без ломающих изменений
- `2.0.0` — ломающие изменения

## Мини-чеклист перед публикацией

1. Изменения завершены.
2. [config/version.php](/Applications/MAMP/htdocs/FIREBALL_CMS/config/version.php:1) обновлён.
3. Проверены нужные файлы в GitHub Desktop.
4. Сделан `commit`.
5. Сделан `push`.
6. Изменения запушены в удалённую ветку.
7. У клиента на странице `Обновление` указан правильный репозиторий.

## Мини-чеклист перед обновлением у клиента

1. Есть свежие коммиты в удалённой ветке GitHub.
2. Изменения уже запушены в нужную ветку GitHub.
3. Репозиторий и ветка в настройках обновления заполнены правильно.
4. Если проект развернут как git-репозиторий, на сервере клиента доступен `git`.
5. Для git-установки в рабочем дереве нет локальных изменений.
6. Для ZIP-установки доступны запись в корень проекта, `config` и `tmp`, а также `ZipArchive`.
7. Нажать `Проверить обновления`.
8. Нажать `Обновить`.
