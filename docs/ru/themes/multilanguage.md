# Мультиязычность

```php
<html lang="<?= htmlSC(current_locale()) ?>">

<?php foreach (available_locales() as $locale): ?>
    <a href="<?= htmlSC($locale['url']) ?>" <?= $locale['active'] ? 'aria-current="page"' : '' ?>>
        <?= htmlSC($locale['title']) ?>
    </a>
<?php endforeach; ?>
```

`switch_locale_url($code)` сохраняет текущий путь и query string. Не добавляйте языковой префикс вручную: базовый язык его не использует.

## Уровни локализации

FIREBALL CMS разделяет три независимых уровня языка:

- `Default Site Locale` хранится в настройках сайта (`default_locale`) и меняется только из админ-панели. Он используется для новых посетителей и как второй уровень fallback.
- `User Locale` хранится в профиле пользователя, а для гостей - в cookie/session. Его меняет только сам пользователь через публичный переключатель языка.
- `Admin Locale` хранится отдельно от публичного языка пользователя и применяется к административной панели.

Публичный переключатель языка не меняет config, `.env` и настройки сайта. Для базового языка CMS добавляет query-параметр `locale`, чтобы middleware мог сохранить выбор текущего пользователя даже без языкового префикса в URL.

## Порядок определения языка

Текущая локаль выбирается на каждый HTTP-запрос отдельно в таком порядке:

1. язык из URL или query-параметра `locale`;
2. язык пользователя;
3. язык из cookie;
4. язык из session;
5. язык сайта по умолчанию;
6. fallback locale.

Для шаблонов используйте `current_locale()`, `site_default_locale()`, `fallback_locale()` и `available_locales()`. Не читайте `app()->get('lang')` напрямую в темах и плагинах.

## Fallback переводов

Языковые файлы ядра и зарегистрированных плагинов загружаются fallback-first: сначала общий fallback, затем язык сайта, затем текущий язык. Более точный язык переопределяет предыдущие значения.

Для данных из таблиц используйте общий helper:

```php
$title = localized_value($row, 'title', 'title');
```

Для SQL-выборок используйте `\FBL\Localization::localizedSql($baseColumn, $originalColumn, $alias, $availableColumns)`. Порядок выбора значения: текущий язык -> язык сайта -> fallback -> оригинальное поле.

## Локализованный кэш

Любые данные, зависящие от языка, должны иметь локаль в ключе:

```php
$cacheKey = locale_cache_key('pages', 'menu:header');
$menu = cache()->get($cacheKey);
```

Не используйте общие ключи вроде `categories`, `menu`, `pages` для публичного локализованного вывода. Для HTML-кэша действует то же правило: один язык - одно cache namespace.

При изменении страниц, статей, категорий, меню, переводов, FAQ или базы знаний очищайте соответствующий публичный кэш. Если используется versioned cache, достаточно поднять version key: старые языковые записи станут недоступными.

## Диагностика

В режиме `DEBUG` локализация пишет в лог:

- текущую, site, session, cookie и user locale;
- источник выбранного языка;
- выбранные ключи переводов;
- используемые cache key;
- ошибки сохранения user/admin locale.

Это помогает проверить, что первый посетитель после очистки кэша не формирует общий HTML или данные для остальных пользователей.
