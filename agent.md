Исправить синхронизацию состава тарифа с уже существующими VPN-подписками в плагине VPN Manager V2.

Текущая проблема:

1. В VPN Manager V2 существует тариф.
2. По этому тарифу уже созданы пользовательские подписки.
3. Администратор добавляет в тариф новый сервер и inbound.
4. Новая связка сохраняется в тарифе.
5. В существующих подписках новый сервер не появляется.
6. Новый subscription node не создаётся.
7. Клиент на новом сервере в 3x-ui не создаётся.
8. Существующая subscription-ссылка продолжает возвращать только старые серверы.
9. В кабинете «Мой VPN» новый сервер не отображается.
10. После обновления подписки в VPN-приложении новый сервер также не появляется.

Нужно реализовать полную цепочку:

добавление server + inbound в тариф
→ определение существующих подписок тарифа
→ поиск отсутствующих подключений
→ создание локальных subscription nodes
→ создание клиентов в соответствующих inbound 3x-ui
→ повторное чтение и подтверждение клиентов
→ обновление revision подписки
→ очистка кэша
→ обновление subscription endpoint
→ отображение нового сервера в «Мой VPN»
→ появление сервера в VPN-приложении после обновления прежней subscription-ссылки

Все изменения должны находиться внутри плагина:

plugins/vpn-manager-v2

За пределами плагина можно только использовать существующие API и компоненты CMS. Не копировать общие сервисы CMS внутрь плагина.

==================================================
3. ПРАВИЛА РАСПРОСТРАНЕНИЯ ИЗМЕНЕНИЙ ТАРИФА
   ==================================================

При добавлении новой активной связки:

plan
→ server
→ inbound

новый сервер должен добавляться в существующие обслуживаемые подписки этого тарифа.

Автоматически обрабатывать подписки со статусами:

- active;
- provisioning;
- provisioning_failed;
- partial_sync;
- sync_error;
- suspended.

Для suspended-подписки новый клиент должен создаваться в 3x-ui с:

enable = false

Не добавлять новый сервер автоматически в подписки со статусами:

- expired;
- cancelled;
- deleting;
- delete_failed;
- deleted.

Если проект использует другие названия статусов, сопоставить их по смыслу.

Истёкшая подписка должна получить недостающие подключения при последующем продлении или повторной активации.

==================================================
4. СОЗДАТЬ СЕРВИС СОГЛАСОВАНИЯ
   ==================================================

Создать отдельный сервис:

VpnPlanSubscriptionReconciler

Допустимые методы:

- previewPlanReconciliation(int $planId): ReconcilePreview;
- reconcilePlan(int $planId, array $options = []): ReconcileResult;
- reconcileSubscription(int $subscriptionId): ReconcileResult;
- findMissingNodes(int $subscriptionId): array;
- findObsoleteNodes(int $subscriptionId): array;
- provisionMissingNode(int $subscriptionId, int $planNodeId): NodeProvisionResult;
- retryFailedNode(int $nodeId): NodeProvisionResult.

Названия DTO и методов адаптировать под реальные соглашения проекта.

Контроллер тарифа не должен содержать основную бизнес-логику.

Контроллер должен только:

1. Проверить permission.
2. Проверить CSRF.
3. Валидировать входные данные.
4. Сохранить тариф и plan nodes.
5. Определить изменения.
6. Запустить reconciliation или фоновую задачу.
7. Показать фактический результат через flash alert.

==================================================
5. ВЫЧИСЛЯТЬ DIFF ТАРИФА
   ==================================================

До сохранения изменений получить старый набор активных plan nodes.

После сохранения получить новый набор.

Определить:

- addedPlanNodes;
- removedPlanNodes;
- unchangedPlanNodes;
- changedPlanNodes.

Для новой связки учитывать:

- server_id;
- inbound_id;
- flow_override;
- is_enabled.

Автоматическое добавление серверов в существующие подписки выполнять только для новых активных plan nodes.

Изменение Flow существующей связки не считать добавлением нового сервера. Для него использовать отдельную логику обновления существующих клиентов.

Удаление plan node не должно автоматически удалять действующие подключения без отдельного подтверждения.

==================================================
6. ПОИСК ОТСУТСТВУЮЩИХ ПОДКЛЮЧЕНИЙ
   ==================================================

Для каждой подписки сравнить:

активные vpn_v2_plan_nodes тарифа

с:

vpn_v2_subscription_nodes конкретной подписки.

Логическая уникальная связка:

subscription_id
+ server_id
+ inbound_id

Новый subscription node требуется только тогда, когда:

- plan node активен;
- сервер активен;
- inbound активен;
- подписка относится к этому тарифу;
- соответствующий node отсутствует;
- нет node в состоянии creating для этой же связки;
- нет node в состоянии create_failed для этой же связки, который необходимо повторить;
- нет активной фоновой операции создания этой же связки.

Если существует node со статусом create_failed:

- не создавать новую запись;
- повторно использовать существующий node;
- повторить provisioning после проверки фактического состояния 3x-ui.

Если существует node со статусом creating:

- сначала проверить, появился ли клиент в 3x-ui;
- если клиент существует — подтвердить node;
- если клиент отсутствует и операция зависла — перевести node в create_failed либо безопасно повторить создание.

==================================================
7. ЗАЩИТА ОТ ДУБЛЕЙ
   ==================================================

Проверить наличие уникального ограничения:

UNIQUE (
subscription_id,
server_id,
inbound_id
)

для таблицы:

vpn_v2_subscription_nodes

Если уникального индекса нет, создать безопасную миграцию.

До добавления индекса проверить существующие данные на дубли.

Если дубли существуют:

1. Не удалять их автоматически вслепую.
2. Проверить соответствующие клиенты в 3x-ui.
3. Определить основную локальную запись.
4. Сформировать диагностический отчёт.
5. Удаление лишнего клиента выполнять только после подтверждения.
6. Не применять уникальный индекс, пока конфликтующие данные не обработаны безопасно.

Использовать блокировки:

vpn_v2:plan_reconcile:{plan_id}

и:

vpn_v2:subscription_node_create:{subscription_id}:{server_id}:{inbound_id}

Повторное сохранение тарифа и параллельные job не должны создавать дубли.

==================================================
8. LOCAL-FIRST PROVISIONING
   ==================================================

Для каждого отсутствующего подключения использовать следующий порядок:

1. Получить подписку.
2. Получить plan node.
3. Получить сервер.
4. Получить inbound.
5. Проверить активность всех сущностей.
6. Определить protocol.
7. Определить network.
8. Определить security.
9. Определить Flow через VpnFlowResolver.
10. Проверить совместимость Flow.
11. Создать локальный vpn_v2_subscription_node.
12. Установить status = creating.
13. Создать UUID только для нового node.
14. Создать уникальный технический client_email.
15. Создать client_sub_id, если он используется.
16. Сохранить текущие:
    - expires_at;
    - traffic_limit_bytes;
    - device_limit;
    - enable-состояние.
17. Зафиксировать локальную транзакцию.
18. После commit выполнить запрос к 3x-ui.
19. Создать клиента через существующий VpnProvisioningService.
20. Повторно прочитать inbound.
21. Найти созданного клиента.
22. Подтвердить его параметры.
23. Только после подтверждения перевести node в active или disabled.

Не удалять локальный node при временной ошибке 3x-ui.

При ошибке:

- node.status = create_failed;
- node.last_error = безопасное сообщение;
- subscription.status = partial_sync или sync_error;
- записать событие;
- предоставить действие «Повторить создание».

==================================================
9. СОЗДАНИЕ КЛИЕНТА В 3X-UI
   ==================================================

Не создавать отдельный второй механизм работы с 3x-ui.

Использовать текущие:

- ThreeXuiClient;
- VpnProvisioningService;
- VpnFlowResolver.

Payload нового клиента должен учитывать:

- client_uuid;
- client_email;
- client_sub_id;
- enable;
- expiryTime;
- totalGB;
- limitIp;
- flow;
- protocol.

Для active-подписки:

enable = true

Для suspended-подписки:

enable = false

expiryTime должен соответствовать текущему expires_at подписки.

totalGB должен соответствовать текущему лимиту подписки.

limitIp должен соответствовать текущему device_limit подписки.

Flow должен определяться по фактическим:

- protocol;
- network;
- security;
- stream settings;
- flow override.

Не определять Flow по имени сервера или remark.

==================================================
10. ПОДТВЕРЖДЕНИЕ СОЗДАНИЯ
    ==================================================

После addClient():

1. Повторно запросить правильный inbound.
2. Найти клиента по точным идентификаторам.
3. Сначала искать по client_uuid.
4. Затем использовать remote_client_id, если он стабилен.
5. Затем client_email.
6. Затем client_sub_id.

Подтвердить:

- UUID;
- email;
- enable;
- expiryTime;
- totalGB;
- limitIp;
- Flow.

HTTP 200 без клиента в повторном ответе считать ошибкой.

Если клиент создан, но Flow не сохранился:

- node.status = sync_error;
- сохранить безопасную ошибку;
- не включать node в subscription endpoint;
- показать администратору:

«Клиент создан, но параметр Flow не был сохранён в 3x-ui.»

Если клиент уже существует:

- не создавать второй;
- сопоставить его только при однозначном совпадении;
- сохранить подтверждённое удалённое состояние.

==================================================
11. ИДЕМПОТЕНТНОСТЬ
    ==================================================

Reconciliation должен быть полностью идемпотентным.

Повторный запуск после успешного выполнения не должен:

- создавать новые локальные nodes;
- создавать новых клиентов;
- менять UUID;
- менять client_email;
- менять subscription token;
- сбрасывать трафик;
- повторно увеличивать revision;
- создавать повторные события успешного добавления без изменения.

Если расхождений нет, результат:

«Подписки уже соответствуют тарифу.»

Revision увеличивать только при фактическом изменении рабочего состава subscription nodes.

==================================================
12. ФОНОВОЕ ЗАДАНИЕ
    ==================================================

Если тариф используется большим количеством подписок, reconciliation не выполнять полностью внутри одного HTTP-запроса.

Создать:

VpnV2ReconcilePlanSubscriptionsJob

Параметры:

- plan_id;
- operation_id;
- initiated_by;
- cursor или last_subscription_id;
- batch_size;
- options.

Job должен:

- обрабатывать подписки пакетами;
- использовать lock;
- сохранять прогресс;
- продолжать после ошибки одной подписки;
- не создавать дубли;
- быть идемпотентным;
- поддерживать безопасный повтор;
- иметь контролируемый timeout;
- не блокировать сохранение тарифа.

Статусы операции:

- pending;
- running;
- completed;
- completed_with_errors;
- failed;
- cancelled.

Показывать в админке:

- всего подписок;
- обработано;
- успешно;
- с ошибкой;
- пропущено;
- время начала;
- время завершения.

Если система уже имеет универсальную таблицу фоновых операций, использовать её.

==================================================
13. ПОВЕДЕНИЕ ФОРМЫ ТАРИФА
    ==================================================

При добавлении нового server + inbound показать настройку:

[✓] Добавить новый сервер в существующие подписки этого тарифа

Рядом показать количество затронутых подписок:

«Новый сервер будет добавлен в 28 действующих подписок.»

По умолчанию настройку можно включать, если это соответствует текущей бизнес-логике.

Если затронутых подписок много:

- сохранить тариф;
- поставить reconciliation в очередь;
- показать flash alert:

«Тариф сохранён. Добавление сервера в существующие подписки запущено.»

Если подписок мало и архитектура допускает синхронную обработку, всё равно использовать единый reconciliation service.

==================================================
14. РУЧНОЕ СОГЛАСОВАНИЕ
    ==================================================

В карточке тарифа добавить действия:

- Проверить расхождения;
- Синхронизировать подписки с тарифом.

«Проверить расхождения» должно работать в dry-run режиме.

Dry-run ничего не изменяет и показывает:

- сколько подписок проверено;
- сколько отсутствующих подключений найдено;
- сколько лишних подключений найдено;
- сколько подписок полностью соответствуют тарифу;
- какие серверы недоступны;
- какие inbound отключены;
- какие данные конфликтуют.

После dry-run администратор может подтвердить reconciliation.

Все изменяющие действия:

- только POST;
- с CSRF;
- с permission;
- без запуска через GET.

==================================================
15. РУЧНОЕ СОЗДАНИЕ ОТСУТСТВУЮЩЕГО NODE
    ==================================================

В карточке подписки показывать:

По тарифу: 3 сервера
Создано: 2 подключения
Отсутствует: 1 подключение

Добавить действие:

«Создать отсутствующее подключение»

Оно должно:

1. Проверить permission.
2. Проверить CSRF.
3. Найти конкретный отсутствующий plan node.
4. Использовать VpnPlanSubscriptionReconciler.
5. Создать local node.
6. Создать клиента в 3x-ui.
7. Подтвердить результат.
8. Увеличить revision.
9. Инвалидировать кэш.
10. Показать flash alert.

==================================================
16. ОБНОВЛЕНИЕ REVISION И КЭША
    ==================================================

После успешного создания хотя бы одного подтверждённого node:

1. Увеличить subscription.revision.
2. Обновить subscription.config_updated_at.
3. Инвалидировать кэш subscription endpoint.
4. Инвалидировать кэш кабинета «Мой VPN».
5. Инвалидировать кэш административного обзора.
6. Инвалидировать кэш списка подключений.
7. Обновить Last-Modified.
8. Обеспечить новый ETag.

Не увеличивать revision до подтверждённого создания клиента.

Не увеличивать revision для create_failed node.

Не очищать весь кэш CMS.

Использовать точечные ключи по:

- subscription_id;
- user_id;
- revision;
- plan_id.

==================================================
17. SUBSCRIPTION ENDPOINT
    ==================================================

Проверить публичный subscription endpoint.

Он должен каждый раз строить конфигурацию из актуальных подтверждённых:

vpn_v2_subscription_nodes

Не использовать старый snapshot списка серверов как источник истины.

В рабочую выдачу включать только nodes, которые:

- подтверждены в 3x-ui;
- имеют корректный UUID;
- имеют корректную конфигурацию;
- находятся в разрешённом рабочем статусе.

Не включать:

- creating;
- create_failed;
- sync_error;
- deleting;
- delete_failed;
- неподтверждённые nodes.

После успешного добавления нового сервера:

- прежний subscription token остаётся действующим;
- URL остаётся прежним;
- QR остаётся прежним;
- ETag изменяется;
- Last-Modified изменяется;
- ответ содержит старые и новые серверы.

Если клиент присылает старый ETag, endpoint не должен ошибочно возвращать 304 после изменения revision.

==================================================
18. КАБИНЕТ «МОЙ VPN»
    ==================================================

После успешного добавления node кабинет пользователя должен показывать:

- новый сервер;
- флаг;
- страну;
- название;
- статус;
- общее количество доступных серверов;
- актуальное количество подключений;
- ту же subscription-ссылку;
- тот же QR-код.

Не показывать сервер как доступный до подтверждённого создания клиента.

Если node создаётся:

«Новый сервер добавляется.»

Если создание завершилось ошибкой:

«Один из серверов временно недоступен.»

Не показывать пользователю:

- текст API-ошибки;
- remote inbound ID;
- полный UUID;
- client_email;
- внутренний статус provisioning;
- технический JSON.

Добавить или сохранить подсказку:

Проверять ownership:

subscription.user_id === currentUser.id

Пользователь не должен видеть чужую подписку.

==================================================
19. СИНХРОНИЗАЦИЯ ТРАФИКА НОВОГО NODE
    ==================================================

Новый node после успешного создания должен автоматически участвовать в общей синхронизации трафика.

Проверить, что VpnV2SyncTrafficJob выбирает новый node.

После первой синхронизации должны обновляться:

- upload_bytes;
- download_bytes;
- traffic_used_bytes;
- traffic_synced_at;
- traffic_sync_status.

Трафик нового node должен входить в общий трафик подписки.

Повторная синхронизация не должна удваивать абсолютный счётчик.

Не сбрасывать трафик существующих nodes при добавлении нового сервера.

==================================================
20. УДАЛЕНИЕ SERVER + INBOUND ИЗ ТАРИФА
    ==================================================

Удаление связки из тарифа не должно автоматически удалять рабочие клиентские подключения.

Безопасная политика:

- добавление plan node распространяется на существующие подписки;
- удаление plan node помечает существующие subscription nodes как obsolete;
- администратору показывается количество затронутых подписок;
- удалённые клиенты продолжают работать до отдельного подтверждённого действия.

Добавить отдельное действие:

«Удалить сервер из существующих подписок»

Перед выполнением показать:

- количество подписок;
- количество удалённых клиентов;
- какие серверы и inbound будут затронуты.

После подтверждения использовать безопасную цепочку:

1. Установить operation status.
2. Удалить клиента из правильного inbound 3x-ui.
3. Повторно прочитать inbound.
4. Подтвердить отсутствие клиента.
5. Удалить или архивировать локальный node.
6. Увеличить revision.
7. Очистить кэш.
8. Обновить subscription endpoint.

Если удалённый клиент не удалился:

- локальный node не удалять;
- status = delete_failed;
- предоставить действие «Повторить удаление».

21. ЧАСТИЧНЫЕ ОШИБКИ

- не откатывать успешно созданных клиентов;
- проблемные nodes оставить create_failed или sync_error;

- Сервер недоступен.

29. ПРИЁМОЧНЫЙ ТЕСТ: НЕДОСТУПНЫЙ СЕРВЕР
    ==================================================

2. Запустить reconciliation.
3. Тариф должен сохраниться.
4. Локальный node B должен остаться с create_failed.
5. Рабочий node A не должен измениться.
6. Subscription endpoint не должен отдавать node B.
7. Пользовательский кабинет не должен показывать B как рабочий.
8. Восстановить сервер B.
9. Запустить повтор создания.
10. Использовать существующий create_failed node.
11. Не создавать второй локальный node.
12. Не создавать второго клиента.
13. После подтверждения установить active.
14. Увеличить revision.
15. Новый сервер должен появиться в subscription endpoint.

6. Revision не должен увеличиваться.
7. Кэш не должен очищаться без изменения.
8. Результат должен сообщить об отсутствии расхождений.

31. ПРИЁМОЧНЫЙ ТЕСТ: ПАРАЛЛЕЛЬНЫЙ ЗАПУСК
    ==================================================

1. Запустить две reconciliation-операции одновременно.
2. Lock должен разрешить только безопасную обработку.
3. Не должны появиться дубли nodes.
4. Не должны появиться дубли клиентов.
5. Операции должны завершиться с корректным статусом.
6. Lock должен освободиться после исключения.

32. ПРИЁМОЧНЫЙ ТЕСТ: КЭШ И ETAG
    ==================================================

3. Добавить сервер.
4. Успешно выполнить reconciliation.
5. Запросить endpoint со старым ETag.
6. Ответ не должен быть 304.
7. Новый ETag должен отличаться.
8. Last-Modified должен обновиться.
9. Новый сервер должен присутствовать.
10. Старый сервер не должен исчезнуть.
11. Token остаётся прежним.

33. ПРИЁМОЧНЫЙ ТЕСТ: ТРАФИК
    ==================================================

2. Добавить сервер B.
3. Выполнить reconciliation.
4. Проверить, что трафик node A не изменился и не сбросился.
5. Использовать сервер B.
6. Запустить синхронизацию трафика.
7. Проверить traffic_used_bytes node B.
8. Проверить общий трафик подписки.
9. Убедиться, что значения A и B не перезаписывают друг друга.
10. Повторная синхронизация не должна удваивать трафик.
